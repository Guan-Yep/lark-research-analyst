# 研报解读专家 (lark-research-analyst v3)

> **前置条件：** 确保已安装 `lark-cli` 并完成 `lark-cli auth login`。本技能依赖 `lark-doc`, `lark-drive`, `lark-whiteboard`, `lark-wiki` 等基础命令。

## 1. 何时使用本 Skill

- 用户上传 PDF 格式的研报，要求极致深度解读、全量图表提取及知识库体系化沉淀。
- 需要在飞书文档中以**附件卡片**、**高清原始图片**、**动态白板**、**颗粒度级深度文字**四位一体的形式展示成果。

## 2. 核心工作流 (V3 标准)

### Step 1: 极致资源解析 (Advanced Extraction)
- **双轨降级提取策略**：使用内置的 Python 脚本 `scripts/extract_pdf.py` 提取 PDF 纯文本及图片。该脚本具备智能路由能力：**如果系统已安装高精度解析工具 `MinerU`，则优先调用它；如果未安装或调用失败，则自动降级使用内置的 `PyMuPDF` 基础解析引擎。** 这确保了既能享受极致解析质量，又在普通环境下具有 100% 的自包含鲁棒性。
- **元数据美化**：提取 PDF 纯净文件名，自动添加业务图标（如 `📄`, `📊`）。
- **自动化命令序列**：
  ```bash
  # 运行内置的 PDF 智能提取脚本（自动选择最优引擎）
  python skills/lark-research-analyst/scripts/extract_pdf.py "path/to/report.pdf" --output-dir "assets/images" --txt-path "extracted_text.txt" --manifest-path "image_manifest.json"
  
  # 创建飞书知识库文档
  lark-cli wiki +node-create --space-id <目标space_id> --title "📄 <纯净PDF名称>" --obj-type docx
  ```

### Step 2: 颗粒度级分层解读 (Granular Summarization)
作为渐进式总结阅读专家，必须严格按照以下 5 层框架对研报进行分析。每一层生成的内容应存放在独立的 Markdown 块中，并注重**深度推演**与**交叉验证**。禁止空泛总结。必须明确指出具体的风险维度（如入口、执行、生态等）、核心逻辑推导过程、以及作者的隐藏假设。

**Layer 1: 快速浏览 (Quick Browse)**
- **目标**：识别研报的骨架与“意图”。
- **深度要求**：不仅写出主题，还要分析作者撰写此报告的**潜在动机**（如行业预警、技术推广等）。
- **输出格式**：主题、作者立场、战略亮点。

**Layer 2: 逐层加深 (Deep Dive)**
- **目标**：挖掘底层逻辑与风险点。
- **深度要求**：针对提取的重点段落，进行“逻辑拆解”，分析其结论是如何推导出来的。
- **输出格式**：核心逻辑链、隐藏假设、关键术语。

**Layer 3: 削减无用信息 (Noise Reduction)**
- **目标**：提炼纯度，识别“噪音”。
- **深度要求**：识别哪些是“共识性信息”，哪些是“真正的增量信息”。
- **输出格式**：增量信息、噪音/共识、逻辑漏洞。

**Layer 4: 摘要核心信息 (Abstract Core)**
- **目标**：结构化重构与跨节验证。
- **深度要求**：将分散在不同章节的数据进行“对撞”，检查是否存在矛盾。
- **输出格式**：跨节数据矩阵、核心结论 1-3。

**Layer 5: 批判性思考 (Critical Thinking)**
- **目标**：评估价值、逻辑与“落地可能性”。
- **深度要求**：代入企业决策者视角，分析如果采纳作者建议，会面临什么实际困难。
- **输出内容**：论点可证伪性分析、证据质量评估、风险推演、落地阻力分析、反思与启发。

*注：检索时结合 `image_manifest.json`，识别对应 PDF 图片，并使用 `grep -iC 10 "关键词"` 获取足够深度的上下文。*

### Step 3: 多维资产自动化集成
- **图片去重过滤**：在插入图片前，务必使用 seed 模型对提取出的图片进行过滤。识别并剔除名称不同但内容完全相同的冗余图片（如重复的页眉、页脚、Logo等），确保不向文档中插入重复图表。
- **图片集成**：在分层解读的对应段落下方，插入过滤后的有效 PDF 图片。
  ```bash
  lark-cli docs +media-insert --doc <doc_token> --file "assets/images/page6_img1.png" --type image --caption "图6: xxx"
  ```
- **白板建模**：在每一层解读后添加 `<whiteboard type="blank"></whiteboard>` 以占位，随后调用 `lark-cli whiteboard +update`，使用 Mermaid 或 DSL 绘制高阶逻辑模型，可视化呈现分层信息。

### Step 4: 附件卡片化与文档封包
- **附件集成**：**不要在报告正文中显示解析出的枯燥长文本。** 将原始的 PDF 文件上传，并在文档末尾“原始报告全文参考”标题下，以**文件卡片 (File Card)** 形式插入即可。
  ```bash
  lark-cli docs +media-insert --doc <doc_token> --file "path/to/report.pdf" --type file --file-view card --as user
  ```
- **最终检查**：确认文档结构完整、附件卡片可用、图片/白板展示正常。

### Step 5: 结果反馈
- **文档链接**：通过获取创建节点的 URL 或通过搜索文档找到最终地址，将最终的飞书文档链接发送给用户，方便用户直接点击查看。

## 3. 最终文档结构规范
1. **封面/元数据** (标题、标签、解读日期)。
2. **结构化深度解读报告** (Layer 1 - Layer 5)。
3. **可视化组件** (白板插入在对应文字下方)。
4. **关键原始图表** (从 PDF 提取的高清图片，对应分析段落)。
5. **原始报告全文参考** (挂载 PDF 附件文件卡片，不展示全文文本)。

## 4. 约束与自审
- **独立性检查**：确保解析步骤使用内置 Python 脚本（`extract_pdf.py`）进行，其内部已实现了 MinerU 优先、PyMuPDF 兜底的智能双轨降级机制，保证环境鲁棒性。
- **颗粒度检查**：检查报告是否详细列出了所有核心技术维度。
- **体验检查**：确保原始报告以"卡片"形式呈现，而非简单的链接或枯燥长文；文字描述是否与插入的原始报告图片相符。
- **一致性检查**：确认白板、原始图片与解读文字三者逻辑统一。确保从 Layer 1 到 Layer 5 呈现出一种“不断穿透表面”的递进感。
